接口介绍

接口测试一般都是指web接口的测试。

restful http

rest风格的HTTP接口，就是使用http协议的http接口，没有增加额外的东西，restful仅仅是一种接口设计的风格。简单的归纳特点如下：

  1. URL用来定位资源。
  2. http的请求方法用来操作资源。

http的get（获取）、post（新增）、put（更新）、delete（删除）等方法可以用来操作url中定位的资源，而不是在url中带有操作资源的方式，比如 http://www.test.com/update 这种。

WSDL是服务端的文件，用xml编写，用来描述webservice提供的方法，接受的参数，返回的数据等。

为什么不直接使用http，而是要使用webservice？
我的理解是webservice更加规范和易用。因为webservice规定了消息体的格式，自然更加规范。易用是指更加方便客户端调用，比如Java有好多种调用webservice的方式，其中一种是使用wsdl2java（或者是wximport）工具，直接根据webservice的WSDL生成webservice的代理类，Java直接调用这些代理类就行了，就像调用本地的普通类一样，不需要编写http的访问的代码了。

RPC

远程过程调用，就是客户端去调用服务端中的方法。REST和webservice都是RPC。
http常作为RPC的传输数据的协议，也有的RPC框架使用自定义的数据传输协议。
RPC是编程语言无关的，客户端和服务端可以使用不同的编程语言。

RMI是Java中的实现RPC的应用程序编程接口，只支持Java，好处是它可以直接传输Java对象。

URL结构

[scheme:][//authority][path][?query][#fragment]
[scheme:][//host:port][path][?query][#fragment]

path可以有多个，每个用/连接，比如scheme://authority/path1/path2/path3?query#fragment。

query参数可以带有对应的值，也可以不带，如果带对应的值用=表示，如：scheme://authority/path1/path2/path3?id=1#fragment，这里有一个参数id，它的值是1。
query参数可以有多个，每个用&连接。
scheme://authority/path1/path2/path3?id=1&name=mingming&old#fragment
这里有三个参数：id，其值是:1；name，其值是:mingming； old，没有对它赋值，所以它的值是null。

在android中，除了scheme、authority是必须要有的，其它的几个path、query、fragment，它们每一个可以选择性的要或不要，但顺序不能变，比如：
其中"path"可不要：scheme://authority?query#fragment。
其中"path"和"query"可都不要：scheme://authority#fragment。
其中"query"和"fragment"可都不要：scheme://authority/path。
"path","query","fragment"都不要：scheme://authority。

  1. #：书签，快速跳转到网页的某一个地方。
  2. ?：参数。
  3. &：联合多个参数，与。
  4. @：缩放原始图片，@320w_440h_1e_1c，e、c不知道是什么意思，目前只见过1e_1c。

http协议

请求行：请求方法、URL、HTTP版本。
请求头：一些字段。
空行。 请求体：get请求为空，post请求为需要上传的数据，但也可能为空。

常见头部字段

响应：
content-encoding：服务器返回的数据很可能是压缩过的，这个指的就是压缩格式。python的requests能自动解压响应，但是urllib不能，需要手动解压。
set-cookie：客户端应该设置那些cooke。

请求：
accept-encoding：客户端支持的压缩格式。
cookie：客户端请求时带的cookie。 keep-alive：不要让http连接立刻断掉，保持一段时间。

post消息体编码

Content-Type:application/x-www-form-urlencoded：请求体必须是文本形式的键值对

Content-Type:multipart/form-data请求体是键值对，值可以是文本，也可以是二进制文件。它可以用来上传文本和二进制文件的混合的消息，或者同时上传多个二进制文件这种

Content-Type:application/octet-stream：请求体必须是一个二进制文件

Content-Type:application/json请求体必须是一个json文本
Content-Type:application/xml请求体必须是一个xml文本
Content-Type:application/javascript请求体必须是一个javascript文本
Content-Type:text/html请求体必须是一个html文本
Content-Type:text/xml请求体必须是一个xml文本

其实post的消息体格式就分4种（如果算上【空消息体】的话，就是5种），什么html，xml等都是纯文本，只不过是带格式的文本罢了。postman中对于post的消息体看上去就比较清晰，jmeter中的有些混乱。

text/xml和application/xml的区别
text/xml默认使用us-ascii编码，除非在请求头中设置字符集，比如Content-Type: text/xml; charset="utf-8"。
application/xml（首选）默认使用utf-8编码，并且在请求头和xml文件中设置的字符集都可以生效，<?xml version="1.0" encoding="utf-8"?>，我猜测在xml文件中设置的话，优先级应该更高一些。

get与post的区别

get传递的参数放在url中，而post传递的参数放在请求体中。
post比get更安全，因为参数不会被保留在浏览器历史记录里。
get的url可以被浏览器收藏为书签，post不可以。
get只能使用url编码，post可以使用url编码、multipart编码等。
get对传输的数据长度有限制，而post则没有。

无连接无状态

无连接：服务器每次只处理一个http的请求，当请求处理完成后就断开连接。客户端再次发起请求后，需要重新建立连接。在请求头中使用Keep-Alive可以让连接长期有效，但是过了keep-alive的时间后，连接还是会断开。

无状态：服务器对客户端的请求处理没有记忆功能，不知道当前客户端是什么状态。后面的http请求如果需要用到前面的http请求产生的信息，那这些信息需要被重传。

cookie session token

cookie是服务端要求客户端设置的属性，下次客户端再发起请求时会带上这些属性。服务器和客户端之间只传输cookie的名称和值，但是在客户端中还存储有cookie的一些属性，比如domain，指定了访问哪个网站时，才会带上这个cookie，这样就可以避免访问任意网站时都带上了所有cookie。

session是服务器中用来保存客户端状态的，毕竟http是【无状态】的，如果这些属性都在客户端中用cookie的方式存储，则请求会变的非常臃肿。所以一般cookie会携带session id，然后与用户相关的状态都放在服务器的session中。

token是一个令牌，有点类似session id，就是给服务端验证客户端身份用的。token可以保存在请求头的Authorization中，也可以保存在cookie。token可以做到服务器无状态（将token过期时间等信息直接保存在token中），也可以做到服务器有状态（服务器可以继续使用session来保存客户端状态）。
token相对于session（或者说session id）的优势，我还是没看明白，暂时不管。

http认证方式

基本认证：在请求头中携带账号密码加密后的字符串，Authorization: username:password。账号密码自然是加密的，加密的方式可以自己选，比如先用以中算法加密password，再与username拼接后，再用另一种算法加密整个字符串。
摘要认证（digest）是用来替代基本认证的另一种方式，也是在请求头的Authorization属性中加入账号密码及其他信息，没见过，先不管。
表单认证，将用户名密码放到post请求的消息体中。

OAuth认证：用户访问服务提供商（比如知乎这种能用微信登陆的网站），服务提供商将用户引导到数据提供商（比如微信），用户在数据提供商那里登陆后，数据提供商创建用户的令牌给服务提供商，服务提供商根据这个令牌就可以访问该用户再数据提供商那边的一些数据，比如用户再数据提供商那边注册时用的用户名和头像。
这里使用token，大概是因为token可以做到服务器无状态，服务提供商可以通过直接校验token来验证用户是否合法。
