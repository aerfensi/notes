Service

Service注意事项

  1. 必须使用显式启动来启动Service
  2. 一个Activity类能有多个实例，但是一个Service类只能有一个实例
  3. service同样运行在主线程，那它为什么叫后台？Android的后台就是指，运行过程完全不依赖UI

Service启动方式

start

  1. onCreate() → onStartCommand() → running → onDestroy()
  2. 第一次启动时会调用onCreate()和onStartCommand()，之后再使用startService(Intent)启动服务，只调用onStartCommand()，除非把他StopService(Intent)后，再startService(Intent)；
  3. 多次StopService(Intent)只会执行一次onDestroy()，因为他已经被销毁了
  4. onStartCommand(Intent intent, int flags, int startId)的第三个参数是指该方法被调用的次数，也就是说第一次被调用这个传进来的值就是1，后面一次累加，Service运行结束再重新启动后，重新计数
  5. stopself()方法可以用来结束start方式启动的service
  6. stopSelfResult(int startId)方法也能用来结束service，但是只有当这个startId和onStartCommand的startId的值是相同时，才会结束这个Service

bind

  1. onCreate → onBind() → running → onUnbind() → onDestroy()
  2. 多个activity可以绑定同一个service，当所有activity都unbind后，这个service就会被destroy
  3. 一个service可以既被start又被bind，那么在手动结束他时，需要unbind和stop

比如希望一个下载服务可以在后台进行，同时又可以启动一个 Activity 来查看下载进度以及暂停、停止服务，就可以用这种方式。
如果service已经启动，bindService是同步的，如果为启动则是异步的，对于使用aidl的远程Service还是本进程的service都是一样的

bindService方法的第三个参数Context.BIND_AUTO_CREATE的意思是如果要绑定的服务不存在就创建它，同时它还是决定服务是否需要被unbind后才能结束的根本因素。
“如果程序既被start有被bind，那结束它就需要既unbind又stop”，这句话的前提是bindService使用的flag是Context.BIND_AUTO_CREATE，如果这个flag是其它的，比如0，那就直接stop，即可结束服务，而且由于flag是0，所以如果服务未启动是不会被创建的，所以需要先start，而不能直接bind。

IntentService

service也是运行在主线程中的，所以如果在他里面执行耗时的操作也是或报ANR异常的。要解决这个问题可以在service中开一个子线程，或者不使用service而是使用它的子类——IntentService

IntentService的特点

  1. 他会单独创建一个子线程，来处理所有发给他的Intent请求
  2. 这些Intent请求会被放在队列中，依次执行
  3. 客户端的stopService或IntentService的stopSelf都不能停止IntentService，虽然它的onDestory会被调用，但是他并没有停止，它会在所有Intent请求都被处理后自动结束
  4. 我只需要重写IntentService的onHandleIntent方法即可，不用重写其他方法

前台Service

不同于IntentService，前台Service不是一个具体的类，它是Service与notification的组合。
前台service通过notification与用户交互，即使在内存不足的情况下它也不会被杀死。
前台服务必须是start方式启动的服务，（我觉得既start又bind的也可以）。

把一个后台服务变成前台很简单，就是使用startForeground (int id, Notification notification)（一般是在onStartCommand中），第一个参数是notification的id，第二个是要展示的notification对象。
用stopForeground (int flags)或者stopForeground (boolean removeNotification)可以把前台服务变回后台（一般在onDestory中），两者效果一样，但是建议用后者，因为旗标难记，后者的参数就很简单——变为后台时是否删除原先的通知。
