MySQL

mysql中字符集的设置为：列→表→数据库→服务，未指定字符集，则使用上一级的字符集。

  • show variables like "character_set%"可以查看当前session的字符集相关变量。
  • character_set_client：客户端发送的SQL语句的字符集，必须与客户端实际字符集相同，比如powershell就是gbk
  • character_set_connection
  • character_set_result：connection返回给客户端的数据的字符集，必须与客户端的实际字符集相同
  • character_set_database：当前数据库的字符集
  • character_set_server：创建数据库是不指定字符集，就用它

client→connection→server，数据在传递过程中会自动转化为字符集变量中设置的字符集，比如数据从client到connection后，会自动转化成character_set_connection。
故只要客户端的实际字符集和character_set_client、character_set_result相同，就不会出现乱码。
实际一般是将character_set_client、character_set_result、character_set_connection设置成相同的字符集。

oracle nls（national language Support）

oracle中的语言设置。
nls_database_parameters、nls_instance_parameters、nls_session_parameters这三张配置数据库的语言相关的参数。

这三张表都有NLS_LANGUAGE、NLS_TERRITORY（地域），他们将影响数据库客户端（如sqlplus）显示的语言、日期格式等。
如果实例的配置文件中没有设置这两个参数，则nls_instance_parameters中的这两个参数来自nls_database_parameters。系统环境变量中如果没有设置这两个参数，则nls_session_parameters的这两个参数从nls_instance_parameters中读取。最后起作用的是nls_session_parameters中的参数。
nls_database_parameters中的NLS_CHARACTERSET中设置的是数据库的字符编码，也就是dbca创建数据库时选择的编码。

sqlplus中文乱码

NLS_LANG，在Linux中是环境变量，在Windows中是注册表的键（临时修改的话也可以直接定义一个同名的环境变量），该变量定义了会话的语言和编码，格式是<Language>_<Territory>.<Clients Characterset>。

NLS_LANG能够指定sqlplus的显示语言，比如可以将language设置为中文，此时需要将字符集设置为与客户端相同。我用的是Windows的powershell，所以字符集需要设置为ZHS16GBK，如果字符集为AL32UTF8，会导致乱码。
NLS_LANG不能解决select查询结果的乱码。 不管如何设置NLS_LANG，powershell中的sqlplus的提示信息可能是乱码或不乱码，但是sqlplus对于的gbk字符集的数据库的查询结果一定是不乱码的。powershell支持的字符集和数据库的字符集相同，都是gbk。
同理，如果数据库为utf8，那powershell+sqlplus对于数据库的的操作一定会乱码。
Navicat支持多种数据库编码，轻松搞定乱码问题。
