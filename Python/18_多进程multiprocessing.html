子进程获取全局变量

  • 子进程可以直接访问全局变量
  • 子进程拿到的是全局变量的副本，不能改变原先的变量；将变量作为参数传递给子进程，效果也一样
  • 在 Linux 中，子进程中使用 id 函数查看全局变量的 id 竟然与主进程中使用id获得的值是一样的！！！

创建 Process 对象时用的 args 参数是('bob',)，为什么需要一个逗号呢？
args参数会被传入 target 的函数，如果参数是一个序列，则会被拆分成多个对象。
例子中的函数f只接受一个参数，所以要出入的是 'bob'，而不是 'b'，'o'，'b'，也就是说这个字符串不能拆。
而单纯的括号并不能表示这是一个元祖，所以会导致字符串被拆。
解决方法是 ('bob',) 或者 ['bob']。

既然进程间对象不共享，那何时需要加锁？目前已知，多进程操作同一个文件的时候！

RLock是可重入锁(reetrant lock)，和Lock对比：

  1. 相同之处：当某一个进程lock.acquire( )后，直到其释放前，其他所有acquire相同lock的进程将被阻塞（包括自身进程）。
  2. 区别之处：是同一个进程能够不被阻塞的多次调用rlock.acquire( )，同样需要相等次数的release( )才能释放后，其他进程才可以结束acquire的阻塞。

manager

multiprocessing.Manager()可以用来创建manager对象，而manager对象可以用来创建进程间共享的容器
支持的类型有list, dict, Namespace, Lock, RLock, Semaphore, BoundedSemaphore, Condition, Event, Queue, Value和Array
文档中没说这些容器是线程安全的，我姑且认为只有Queue是线程安全的吧

多进程之间不能共享变量，比如我把一个list传入给一个process的target函数，在process中修改这个list，原先的list也不会改变
很奇怪我在process内外分别调用id()函数获得的值居然是一样的，但这确实是两个不同的对象

我认为multiprocessing.Lock被传入子进程后应该也不是原来的那个对象了，但是它确实依旧能起作用，不仅仅是它，还有multiprocessing.Event，multiprocessing.Condition等
既然如此，那manager中的Lock等还有什么意义吗？答案当然是有的

当使用进程池的时候，如果将Lock作为参数传递，pool.apply_async(child_worker, (my_parameter,lock))就会报出如下错误
RuntimeError: Lock objects should only be shared between processes through inheritance
没找到解释，我猜测是【Lock对象只能在父子进程间共享，而进程池中的进程不一定是当前进程创建的】
解决办法当然是使用manager了

进程池 Pool

python并发之multiprocessing

Pool支持with语句，它的__exit__方法似乎是调用了Pool的terminate方法，导致所有进程都死了
如果希望对Pool使用with语句的话，最好还是配合使用contextlib.closing()函数

map(func, iterable[, chunksize])：相当于内置的map函数，同步进程，执行完后返回函数f return的结果，只接受一个iterable作为函数f的参数
map_async(func, iterable[, chunksize[, callback[, error_callback]]])：异步，立即返回；callback是一个接受一个参数的回调函数，当func执行完毕后，func的返回值会传给callback并执行callback
apply()：同步，能传给函数的参数是*args,**kwargs
apply_async()：异步
join()：阻塞当前进程，等进程池执行完毕
close()：阻止再提交进程，但是不会影响现有进程的运行
terminate()：退出进程池，强制停止所有进程

上面立即返回的异步函数返回的是一个multiprocessing.pool.AsyncResult类型的对象。类似于concurrent.futures.Future类型

因为进程池中的进程能返回执行结果，所以进程池是获取子进程的结果的一种方法
