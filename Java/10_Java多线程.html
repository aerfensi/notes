synchronized

synchronized 可以用来修饰方法、静态方法、代码块。
可以保证代码的原子性与可见性。
原子性：原子性就是不可分割，代码块在执行时，CPU不能中断当前线程切换到其他线程。
可见性：操作执行完毕后立刻将改变的变量值刷新到主内存，保证其他线程获取到的都是最新的值。

volatile 关键字用于修饰变量，只能保证可见性，因此 synchronized 关键字为首选。

当一个线程访问 synchronized 修饰的代码前必须获得锁，否则改线程会一直等待，代码退出或抛出异常后必然会释放锁。

方法和代码块的锁即为该方法或代码块所属对象。也就是同一个对象中的多个不同 synchronized 修饰的方法和代码块的锁是相同的，一条线程在访问其中一个 synchronized 方法时，其他线程不能访问该独享的其他 synchronized 方法。

静态方法的锁是当前对象的 Class 对象（所有同一个类的对象的 Class 对象都是同一个），所以含有静态 synchronized 方法和普通 synchronized 方法的对象中的这两种方法的锁是不同的，这两种方法可以同时被两个线程访问。

Java 的 synchronized 块是可重入的，可重入的意思是线程可以重复获得它已经持有的锁，这样一来，相同锁的方法可以互相调用，而不会发生死锁。

线程间通信，wait、notify、notifyAll

这三个方法必须在同步块或同步方法中调用。
通信的原理是多个线程都需要访问同一个对象时，该对象内部通过调用这三个方法来使线程等待或唤醒线程。
对象拥有线等待池和锁池、访问对象的线程可以在等待池中等待（对象调用 wait 方法），在锁池中竞争锁（对象调用 notify 方法）或者在获得锁后执行。

notify会唤醒所有在该对象等待池中的线程，而notify只会随机唤醒一个。优先级高的线程有更大的可能竞争到锁，而不是有更大的可能被唤醒。

wait(long millis)可以使线程在超过一定时间后自动唤醒，而不需要 notify 方法。
wait 方法和 Thread.sleep 方法最大的不同是 wait 会释放锁。

通知遗漏问题，早期通知问题

所谓通知遗漏就是当线程 A 还没有 wait 的时候，线程 B 已经使用了 notify 方法，这样当线程 A wait 后便没有线程来唤醒它了，那它将一直等待。

所谓早期通知问题就是线程被 notify 方法唤醒，但其实该线程还有等待的其他条件没被满足。比如线程需要从 list 中取出一个对象，某个时间点它被唤醒，但此时 list 为空，无法取出对象。

以下是解决通知遗漏问题和早期通知问题后的代码示例。

通知遗漏问题（需要在方法A中加入sleep来保证方法B先被执行）。
main 方法中启动两条线程，一条执行方法 A，一条执行方法 B。
正确的顺序应该是，线程 A 等待添加产品 -- 线程 B 添加产品 -- 线程 A取出产品。
如果线程 B 先添加产品、那线程 A 将一直等待添加产品，它并不知道产品已被添加。
所以线程A在等待前要判断是否真的需要等待，比如生产者消费者模型，消费者要在当前产品数为0的情况下才wait。

早期通知问题（需要在方法B中添加sleep来保证方法A先被执行）。
main 中启动两条消费者线程都执行方法A，再启动一条生产者线程来执行方法B。
开始时有两条消费者线程在等待产品添加，之后生产者线程添加产品后，其中一条消费者线程被唤醒，取出产品（现在已经没有产品了），接着另一条消费者线程被唤醒，该线程没有产品可取，所以它应该继续等待。
消费者线程在被唤醒后，先判断是否有产品，有就消费，没有就继续等待。

方法是在 wait 外面包裹 while 循环，如果仅仅是针对通知遗漏问题，那 wait 外面包裹 if 判断也没什么问题。

lock 锁

lock 可以用来代替 synchronized 关键字，lock 接口有三个实现类 ReentrantLock（可重入锁）、ReetrantReadWriteLock.ReadLock（可重入读锁） 和 ReetrantReadWriteLock.WriteLock（可重入写锁）。

使用 synchronized 所有同步代码块都是互斥的，但是使用读写所的话，两个读锁不是互斥的，既然都是读取数据那也就没有互斥的必要。

lock 的优势有：

一、等待可中断，synchronized 等待不可中断
这里的等待不是指对象调用 wait 方法后，线程进入对象的等待池，而是指线程在锁池内等待其他线程释放锁。
可以调用lock 对象的 lockInterruptibly() 方法获取可中断的锁。如果某个对象中的同步代码块获得的锁是可中断的，有线程在执行该对象的同步代码块时没有获取锁，那该线程将等待背的线程释放锁，此时其他线程可以调用那个正在等待的线程（也就是 thread 对象）的 interrupt() 方法来中断它的等待，该线程将放弃执行同步代码块，继续执行下面的代码。

二、 可以实现公平锁
创建锁时传入 true 参数，new ReentrantLock(ture)。线程在竞争同一个锁时按照申请时间排队等待。

三、同一个锁可以绑定多可条件
Condition condition = lock.newCondition()，condition 对象可以使用 await(), signal(), signalAll() 方法，效果同 object 的 wait(), notify(), notifyAll()，当然，只能由同一个。condition 对象的 signal 去唤醒 同一个 condition 对象的 await，lock 对象可以创建多个 condition 对象。

线程任务 Runnable 和 Callable

Java中的Runnable、Callable、Future、FutureTask的区别与示例

  • Runnable：无返回值，无法抛出异常，仅仅需要实现一个run方法
  • Callable：和Runnable差不多，仅仅需要实现一个call方法，有返回值，能抛出异常。一般被ExecutorService执行
  • Future：代表Runnable或者Callable任务的执行结果，可以用来取得Callable的返回值，取消任务的执行，判断任务是否被取消，判断任务是否执行结束。ExecutorService的submit方法可以返回Future对象
  • FutureTask：实现Runnable和Future接口，同时构造器可以接受Runnable对象或者Callable对象。它可以被Thread和ExecutorService执行

Future的get方法可获得运行的返回值，当get方法被调用时，当前线程被阻塞，直至Callable的call方法执行完毕并返回

Thread

线程的六种状态

  • NEW：线程被创建但是还没有启动
  • RUNNABLE：线程可运行（条件具备，就差 CPU），或者线程正在运行
  • BLOCKED：阻塞，正在等待其他线程释放锁
  • WAITING：正在等待其他线程唤醒它
  • TIMED_WAITING：定时等待，到达时间后，即使没有被其他线程唤醒，也将自己醒来
  • TERMINATED：终止

进程的状态

  • 就绪：条件具备，就差 CPU
  • 运行
  • 阻塞
  • 挂起：进程挂起后进入【就绪】状态

线程中断

thread 对象的 interrupt 方法可以用来中断该线程，其实只是个给线程设置一个标志位，对线程的执行没有影响。
isInterrupted 方法判断线程有没有设置中断标志位。
interrupted 方法判断该线程有没有设置中断标志位，判断过后清除中断标志位。

任何能抛出 InterruptedException 异常的方法都能被中断，比如可以用 interrupt 中断线程的 sleep，sleep 被中断的话，中断标志位也会被清除。

线程池

Executor线程池详解
Android concurrent（一）—— Executor接口

线程池内有多个未死亡的线程（Thread对象），当一个任务需要用该线程池中的线程执行时，如果有线程处于等待状态，该任务就会被执行，如果所有线程都处于运行状态，则该任务在队列中等待有线程执行完毕。
应该就是线程池内有一些 Thread 对象，然后可以把 Runnable 不断地传给 Thread 去执行。

  • 任务：需要执行的一系列操作，比如Runnable接口的对象
  • 线程：任务的执行者，一个任务结束后，线程可以等待下一个任务，也可以被销毁

线程池的生命周期：运行（创建之后）、关闭（shutdown 方法被调用后）、终止 线程池必须要被 shutdown，否则的话它将一直等待新任务，程序会一直运行

线程池的几个常用类

  • Executor：线程池的顶级接口、实际使用中很少用到
  • ExecutorService：真正会用到的线程池接口
  • Executors：线程池的工厂类，用来创建一些常用的线程池
  • ThreadPoolExecutor、ScheduledThreadPoolExecutor：用来自定义一些线程池

Executor（顶级接口）、.*ExecutorService（继承自顶级线程池的其他线程池接口、抽象类），.*Executor（线程池的具体实现类）

JDK 文档强烈建议使用工厂类来创建常用的线程池对象，而不是手动 new

创建线程池

Executors（普通类）：包含几个静态方法，用来创建常用的线程池。这些方法返回的对象是ExecutorService（接口）类型的。

  • newSingleThreadExecutor：永远只有一条线程在运行，也就是使用该线程池的任务都要排队。如果唯一的线程挂了，会自动创建一个新的线程
  • newFixedThreadPool：创建固定大小的线程池。每次提交一个任务就创建一个线程，直到线程达到线程池的最大大小
  • newFixedThreadPool：创建一个可缓存的线程池。如果线程池的大小超过了处理任务所需要的线程，那么就会回收部分空闲（60秒不执行任务）的线程
  • newScheduledThreadPool：创建一个大小无限的线程池。此线程池支持定时以及周期性执行任务的需求

手动 new 出线程池

一般用ThreadPoolExecutor（普通类），它实现了ExecutorService接口，并且它有一个子类ScheduledThreadPoolExecutor

ThreadPoolExecutor构造器的参数

  • int corePoolSize：池中所保存的线程数，包括空闲线程。
  • long maximumPoolSize：池中允许的最大线程数。
  • long keepAliveTime：当线程数大于核心时，此为终止前多余的空闲线程等待新任务的最长时间。
  • TimeUnit unit：keepAliveTime 参数的时间单位。
  • BlockingQueue workQueue：执行前用于保持任务的队列。此队列仅保持由 execute方法提交的 Runnable任务。
  • ThreadFactory threadFactory：执行程序创建新线程时使用的工厂。
  • RejectedExecutionHandler handler：由于超出线程范围和队列容量而使执行被阻塞时所使用的处理程序。

Timer 和 TimerTask

Timer：英文意思是【定时器】，一个 Timer 对象就是一条线程。使用 schedule 方法启动线程（不是和 Thread 一样的start方法）。
TimerTask：被 Timer 对象反复执行的任务，其内部的 run 方法需要被重写。
这个两个类都有cancel方法，Timer 用该方法终止线程，该线程中所有 TimerTask 都不会被执行了，TimerTask 用它来把自己从Timer的线程中移除，但是 Timer 的线程不会终止，如果它还有其他 TimerTask，那么这些 TimerTask 依然会被执行。
