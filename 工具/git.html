配置git

git 的配置信息保存在 /etc/gitconfig、~/.gitconfig 和 .git/config 文件中，这几个配置文件的优先级递增。

​x
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.ci commit
git config --global alias.st status
​
# 查看git的全部配置信息
git config --list
# 设置git使用的默认文本编辑器
# --global：写入到 ~/.gitconfig --system：写入到 /etc/gitconfig 什么都不加：写入到 .git/config
git config --global core.editor emacs
# 解决中文文件名乱码
git config --global core.quotepath false

使用远程仓库的方法

情况1：把本地已经有一个项目了，希望把他放到远程仓库里去。

  1. 进入需要上传的项目文件夹，执行git init，这个目录就成为了master分支
  2. 添加远程仓库：git remote add origin https://github.com/aerfensi/backup.git
  3. 第一次push：git push -u origin master 第一次push的时候用【-u】参数，这样他会关联本地master分支与远程仓库分支，以后只需要执行git push，git pull就行了，不用指定远程仓库和分支名。如果远程仓库中已经有些文件了，希望pull下来 git pull origin master

情况2：clone远程仓库的项目。

  1. 直接：git clone https://git.oschina.net/用户名/项目名
  2. 添加远程仓库：git remote add origin git@git.oschina.net:用户名/项目名。git remote -v 可以用来查看当前添加的远程仓库。如果是https开头的，每次pull，push都要输用户名和密码，git@开头的就不需要。
  3. git remote rm 远程仓库别名

git记住账号和密码

  1. 在用户根目录新建文件.git-credentials
  2. 输入https://${用户名}:${密码}@github.com，比如https://duanku:duanku12@github.com
  3. 进入需要使用这个账号密码的 git 工程目录，输入git config --global credential.helper store
  4. 在用户根目录的.gitconfig文件的最后有如下字样，即表示成功
xxxxxxxxxx
[credential]
helper = store

添加更改到暂存区

把文件添加到暂存区：git add file或git add .。
commit之前多次add同一个文件，暂存区中保存的是最新版。
删除暂存区中的文件：git rm file 或 git rm -r 递归删除（提交之后会同时删除工作区中的文件）。
移动或重命名文件：git mv（提交之后会移动或重命名工作区中的文件）。
将所有更改（包括add，rm，mv）都添加到暂存区：git add -A。

提交到版本库

简单提交：git commit -m "提交信息" 工作提交：git commit . 修改上一次的提交信息：git commit --amend 能修改提交的说明，同时如果暂存区里面有新增的修改文件，也会被追加到这笔提交中。

撤销更改

撤销未添加到暂存区的更改：git checkout -- filename （ git checkout . ）表示撤销所有文件。
撤销对暂存区的添加：git reset HEAD [<file>...]。
撤销版本库中的：git reset --soft HEAD^。
撤销工作区、暂存区、版本库里内容：git reset --hard HEAD^。

如果reset命令使用了 --soft 或者 --hard 参数，那就不能在后面加文件了，也就是说只能全部回退，而不能只回退指定的文件 HEAD指向最近一次提交，HEAD^指向上一次提交，HEAD^^上上次提交，HEAD~10，前10次提交；也可以直接使用commit id。

把工作区、暂存区、版本库回退到最近一次 git pull 之前：git reset --hard ORIG_HEAD。
清除目录中新增的文件：git clean -df。

-d是指连同新建的目录一起删除，-f是指强制删除。

回退：git revert commit-id/HEAD，其实就是找出某一笔提交修改的文件和内容，然后删除这些内容，也就是再次修改了这些文件的内容，所以之后需要把这个修改提交，以完成内容回退。

查看提交

查看提交历史：git log [ HEAD或commit id或者 ]。
列出前n次提交：-n。
查看各文件的详细变更：-p。
查看某个文件的历次提交：-- file。
列出每次提交的变更文件：--stat。
查看特定作者的提交：--author。
查看特定提交者的提交（提交者和作者可能不是同一个人）--committer。
查看提交说明中含有特定关键字的提交：--grep。
查看某段时间内的提交：--after="2008-10-01" --before="2008-11-01"。
除了使用具体的日期外，还有另一种表述方式：--after=3.days 三天内的提交；--before=3.days 三天以前的提交；除了用 day，还能使用 week。

显示详细信息：git show。
与log不一样的是，show不加别的参数就只显示最近一次提交的详细信息。

显示变更内容：git diff 比较暂存区与工作区，git diff --cached 比较版本库与暂存区。

git blame -L n +m file：从n行开始到m行，查看该文件的这几行的最后一次修改分别是哪一次提交。

查看更改

查看工作区与暂存区文件的区别：git diff [-- <file>...]（如果暂存区没文件，就是和版本库比）。
查看暂存区与版本库文件的区别：git diff --cached [-- <file>...]。
查看工作区与版本库文件的区别：git diff <commit> [-- <file>...]。

查看两次历史提交中的变化：git diff <commit> <commit> [-- <file>...]。
查看简略信息：可以使用 --stat 参数。

存储更改

在本地有未提交的更改的情况下，git pull 成功下载新的文件（如果不使用 git pull --rebase 的话）。
这种情况可以先把本地更改存储起来：git stash（没错，没有【.】），再 git pull。
git stash list 查看所有存储，stash@{0}表示最近的一次存储，数字越大表示历史越久远。
git stash apply stash@{0} 还原存储的内容，不指定是哪一次存储就会还原最近的一次 如果发生冲突，解决冲突后，add 再 commit 即可。
git stash drop 删除某一个存储。

分支管理

merge和rebase详解

不管是merge还是rebase都只会影响当前分支，其它分支不受影响。
rebase是对原先做出的提交在变基之后重新提交，所以新的提交的commit id是和原先不同的。

查看分支：git branch。
列出各分支及其最后一次提交信息：-v。
仅列出与当前分支合并的分支：--merged。
仅列出未与当前分支合并的分支：--no-merged。
查看远程分支：-a。

新建分支：git branch <branch>。
切换分支：git checkout <branch>。
新建并切换分支：git checkout -b <branch>。
新建分支并与远程分支关联：git checkout <remote_branch> -b <branch>。
切换到远程分支，新建的本地分支直接使用远程分支名：git checkout --track <remote_branch>。

删除分支：git branch -d <branch>。
恢复分支：git branch <branch> <SHA1>（分支删除时会在屏幕上打印该分支的 SHA1 值）。

合并分支：git merge <branch>。
合并分支发生冲突时，手动解决冲突，再add，再commit。

冲突的形式如下： 

xxxxxxxxxx
<<<<<<< HEAD
<div id="footer">contact : email.support@github.com</div>
=======
<div id="footer">
  please contact us at support@github.com
</div>
>>>>>>> iss53

合并分支：git rebase <branch>。
解决冲突后add，再使用git rebase --continue继续，或者使用git rebase --abort放弃。

当前分支：master；需要合并的分支：dev。
merge：把dev的最后一次提交合并过来。
rebase：把dev和master分支的从分支节点到目前为止的所有提交都合并到master上来，所以rebase会产生更多的冲突。

生成与应用patch

patch 就是把 git diff 的内容输出到一个后缀名为 patch 的文件中，所以可以直接使用git diff > file.patch这种形式来生成 patch，当然也可以使用以下命令。

  1. 两个节点之间的提交：git format-patch 节点A 节点B
  2. 单个节点：git format-patch -1 节点A（-n就表示要生成几个节点的提交）
  3. 最近一次提交节点的patch：git format-patch HEAD^依次类推……

git apply patch_file：凡是有冲突的，整个patch都不会合入，用--reject可以强制合入无冲突的部分，有冲突的部分会在目录下生成一个后缀名为rej的同名文件，该文件中保存有冲突的部分，根据需要手动合入。
